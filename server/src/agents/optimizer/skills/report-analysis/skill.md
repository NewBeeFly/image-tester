# 报告分析技能

## 使用场景

当用户询问测试运行结果、失败原因、通过率变化、失败用例为何失败等问题时，加载此技能。

## 工作流程

1. **获取概览**：调用 `get_test_run_report(run_id)` 获取运行统计 + 使用的提示词信息
2. **抽样失败**：调用 `get_run_items(run_id, pass=false, limit=10)` 抽样失败用例
3. **视觉复核（关键）**：从失败用例中挑选 **最多 3 个代表性案例**，收集它们的 `image_absolute_path`，调用 `load_images_base64(file_paths=[...])` 批量加载图片。工具返回 JSON 数组，每个元素有 `base64` 字段。将每张图片的 `base64` 作为独立 image message 传入模型，让模型直接看图判断：
   - 原模型输出是否正确？
   - 失败是因为模型对图片识别错误，还是断言配置过严/变量不匹配？
   - 图片内容与原 `model_output`、`assertion_failures` 是否一致？
4. **模式识别**：将失败用例按失败类型分组（如 jsonPath 不匹配、contains 缺失、格式错误、视觉识别错误等）
5. **趋势分析**（如需）：调用 `compare_runs(run_ids)` 对比多次运行的通过率变化
6. **总结输出**：以结构化格式呈现分析结果，明确指出多少失败属于「模型识别错误」、多少属于「断言/配置问题」

## 数据获取策略

- 失败数量 > 10 时，先抽样 10 条分析模式，告知用户可查看更多
- 默认加载 1-3 张代表性失败图片进行视觉复核，优先选择：
  - 高频失败模式中的典型案例
  - 看起来「模型输出明显与图片不符」的案例
  - 不同失败类型的案例各 1 个
- 图片以独立 image message 传入，不要拼接到 text 内容中
- 按需渐进式获取，避免一次性加载全部数据

## 消息构造规范（多模态分析）

当需要向大模型展示图片时，必须按以下方式构造：

1. 先发送一条 text 消息，包含：
   - 案例基本信息：`case_id`、`image_path`
   - 模型原始输出 `model_output`
   - 断言失败详情 `assertion_failures`
   - 待分析问题：请判断该图片中实际场景是什么？原模型输出是否正确？
2. 再发送一条或多条 image 消息：
   - 解析 `load_images_base64` 返回的 JSON 数组，使用每个元素的 `base64` 字段（即 `data:image/...;base64,...`）作为 image_url
   - type 为 `image_url`
3. 最后发送分析指令 text 消息

严禁把 base64 字符串直接拼在 text 里。

## 输出格式示例

```
## 运行 #109 分析报告

**统计**：通过 158 / 失败 99 / 总计 257 | 通过率 61.5% | 平均耗时 6.3s

**失败根因分类**：
| 根因 | 数量 | 占比 | 典型断言失败 |
|------|------|------|------------|
| 模型对图片识别错误 | 45 | 45% | $.scene_no 期望"10"实际"2" |
| match_reason 缺失 | 30 | 30% | $.match_reason 为空 |
| JSON格式错误 | 24 | 24% | 无法解析模型输出 |

**视觉复核结论**（已抽样 3 张图片）：
- case_id=2048：图片实际为「白色标准大展柜」，模型输出 scene_no=24 属于识别错误
- case_id=2051：图片实际为「A1 复合展台」，模型输出与图片一致，但断言期望 scene_no=1，属于断言/标注问题
- case_id=2077：图片模糊，模型输出 match_reason 为空，属于模型输出质量问题

**建议**：针对「scene_no 识别错误」，建议在 system_prompt 中增加对场景编号的明确说明...
```
